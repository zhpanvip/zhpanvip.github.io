Android 架构思想与 MVVM 框架封装

一、Android 开发中的架构演化

  1. MVC 简介

MVC 是 Model-View-Controller 的简称，它是 Android 原生开发最基础的架构。简单来说MVC是通过Controller去操作Model层的数据，并且返回给View作展示。MVC的三个分层作用如下：

  • 模型层(Model) 负责与数据库和网络层通信，并获取和存储应用的数据。
  • 视图层(View) 负责将 Model 层的数据做可视化的处理，同时处理与用户的交互。View 层包含了所有显示在屏幕上的 View，对应 Android 中的 XML 与 Activity
  • 控制层(Controller) 用于建立Model层与View层的联系，负责处理主要的业务逻辑，并根据用户操作更新 Model 层数据。在 Android 中由 Activity 承担这一职责。

  • 开发时不注重分层，Model层代码也被塞进了Activity/Fragment，使得代码层次更加混乱。

  1. MVP 简介

MVP 是 Model-View-Presenter 的简称。 MVP 将 MVC 的 Controller 改为 Presenter，将逻辑层的代码从 Activity 中抽离到了 Presenter 中，使得代码层次更加清晰。

  • 模型层(Model) 与MVC中的一致，同样是负责与数据库和网络层通信，并获取和存储应用的数据。
  • 视图层(View) 负责将 Model 层的数据做可视化的处理，同时与Presenter层交互。跟 MVC 相比 View 层与Model 层没有耦合。
  • 控制层(Presenter) 主要负责处理业务逻辑，通过接口回调给 View 层刷新UI。

MVP 的结构图如下图所示。

从上图可以看出，View 持有 Presenter 层，当 View 层接收用户操作后会通知 Presenter 去处理业务逻辑。接着 Presenter 层通过 Model 去获取数据，然后 Model 层又将最新的数据传回 Presenter 。 同时，Presenter 层又持有 View 层的引用，进而将数据传给View层进行展示。

  1. MVVM 简介

MVVM 是 Model-View-ViewModel 的简称。这一架构解决了 MVP 中存在的问题。MVVM 是Android 官方近来主推的一个架构模型，同时也是目前 Android 项目的主流架构。

  • 模型层(Model) 与 MVC/MVP 中的 Model 一致，负责与数据库和网络层通信，并获取和存储应用的数据。
  • 视图(View) 负责将 Model 层的数据做可视化的处理，同时与 ViewModel 层交互。
  • 视图模型(ViewModel) 主要负责业务逻辑的处理，同时与 Model 层 和 View层交互。

MVVM 架构的结构图如下。

View 持有 ViewModel，View 产生的事件会交给 ViewModel 处理。而 ViewModel 又会将获取数据的逻辑交给 Model 层。Model 层获取到数据后通知 ViewModel，ViewModel 再通知 View 更新UI。

MVVM 的本质是数据驱动，相比于MVP ，MVVM 实现了 ViewModel 与 View 层的解耦。解耦做的更彻底。

二、使用 Jetpack 组件封装 MVVM 架构

1. Jetpack MVVM 简介

MVVM 是 Google 官方推荐的框架。为此，Google 提供了一系列实现 MVVM 的工具，这些工具都被包含在Jetpack 组件中，例如 LiveData、ViewModel、以及 DataBinding 等。下面是官方给的一个通过ViewModel 、LiveData 等组件实现的 MVVM 架构图。

图中的箭头都是单向的，View层指向了ViewModel层，表示View层持有ViewModel层的引用，但ViewModel层不持有View层。ViewModel层是有Repository层，Repository层不会持有ViewModel。

  • 视图(View) 对应图中的Activity/Fragment，包含了布局文件等于界面相关的东西；
  • 视图模型(ViewModel) 对应图中的Jetpack ViewModel 与LiveData，用于持有与UI相关的数据，且能保证在旋转屏幕后不丢失数据。另外还提供了给View成调用的接口以及和Repository通信的接口；
  • 模型层(Model) 对应图中的 Repository，包含本地数据与服务端数据。
