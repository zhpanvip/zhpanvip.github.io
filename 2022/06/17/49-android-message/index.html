反思 Android 消息机制设计与实现

上篇文章介绍了 Android 中的 Binder 机制。Binder 在 Android 系统中占有着举足轻重的地位，它是 Android 系统中跨进程通信最重要的方式。而另外一个重要的且能与Binder相提并论的角色便是本文要分析的 Handler。Binder 支撑起了 Android 系统进程间的通信，而 Handler 支撑起的则是进程内线程间的通信。同时，Android 应用程序的正常运行皆依靠 Handler 的消息机制驱动，这其中就包括触摸事件的分发、View的绘制流程、屏幕的刷新机制以及Activity的生命周期等等。

关于 Handler 其实早在几年前笔者就写过一篇《追根溯源—— 探究Handler的实现原理》的文章。 但是鉴于当时对于 Handler 的理解并不那么深刻，所以这篇文章的内容与网上大多数写Handler的文章一样仅仅是为了分析源码，并没有写出对Android消息机制深刻的理解和认识。当然，并不是说这样的文章不好，对于初学者来说更适合阅读这样的文章的。所以，如果你对于 Handler 还没有太熟悉的话，不妨先读一读。

而此时，作为一个已有多年Android开发经验的从业者，以及在阅读了大量的 framework 源码之后，对于Android的消息机制有了一些新的和更加深刻的理解，这也是为什么要写这篇文章的原因。

一、从“生产者-消费者”模型说起

关注笔者比较久的同学可能看过我之前写过的一篇文章 《深入理解Java线程的等待与唤醒机制》。在这篇文章中为了分析 synchronized 锁的等待与唤醒机制，举了一个 “生产者-消费者” 问题的例子。“生产者-消费者” 问题又称有限缓冲问题（Bounded-buffer problem），是一个多线程同步问题的经典案例。该问题描述了共享固定大小缓冲区的两个线程——即所谓的“生产者”和“消费者”——在实际运行时会发生的问题。生产者的主要作用是生成一定量的数据放到缓冲区中，然后重复此过程。与此同时，消费者会在缓冲区消耗这些数据。该问题的关键就是要保证生产者不会在缓冲区满时加入数据，消费者也不会在缓冲区中空时消耗数据。其模型图如下：

可以看得出来，图中的 “生产者”和”消费者”处于两个不同的线程，但是他们共用了同一个队列。生产者在完成数据的生产后会通过notify方法唤醒消费者线程，且如果队列满的时候，消费者线程会调用 wait 方法来阻塞自身。同时，消费者线程在被唤醒后则会从队列中取出数据，并通过 notify 方法唤醒生产者线程继续生产数据，当队列中的数据被取空的时候，消费者线程同样会调用 wait 方法阻塞自身。关于这一模型的代码实现在这里就不重复贴出了，有兴趣的可以在《深入理解Java线程的等待与唤醒机制》这篇文章中查看。

其实”生产者-消费者“模型解决的不仅仅是缓冲问题，还有一点重要的是解决了线程间通信的问题。在生产与消费的过程中由于公用了缓冲队列，”生产者“产生的数据从生产者线程传递到了消费者线程，即实现了线程的切换。

关于”生产者-消费者“模型的案例其实在我们平时的开发中随处可见，例如Rxjava的流控制就是典型的”生产者-消费者“模型，除此之外还有Android系统对于输入事件的采集与派发，以及本篇文章的主角Handler都是基于”生产者-消费者“模型来设计的。

了解了“生产者-消费者”模型之后对于Android 消息机制的设计与实现会有很大的帮助。

二、消息机制框架的设计与实现

现在让我们切回 Android，来想一下 Android 中的场景。在文章开头已经提到Android中触摸事件的分发、View的绘制流程、屏幕的刷新机制以及Activity的生命周期等都是基于消息实现的。这意味着在Android应用中随时都在产生大量的 Message，同时也有大量的 Message 被消费掉。

另外我们都知道在Android系统中，UI更新只能在主线程中进行。因此，为了更流畅的页面渲染，所有的耗时操作包括网络请求、文件读写、资源文件的解析等都应该放到子线程中执行。在这一场景下，线程间的通信就显得尤为重要。因为我们需要在网络请求、文件读写或者资源解析后将得到的数据交给主线程去进行页面的渲染。

那在这样的背景下如果你作为系统的设计者，会如何设计并实现 Android 中的消息机制，并能让其满足具有缓冲与线程切换的功能呢？

有了第一章的知识后我想你一定会很自然的想到使用”生产者-消费者“模型来实现。因为这一模型既能解决数据缓冲问题，又能实现线程间的切换。

没错，Android系统的设计者也是这么想的，于是便有了 Handler 这一杰作！接下来让我们跟随Android消息机制设计者们的思维来看一下如何实现消息机制的框架。

2.设计消息机制的”生产者“–Handler

系统各处产生的消息取消存入到消息队列中，等待消费者取出消费。因此我们可以设计一个包装类来实现消息插入到消息队列，我们可以将这个包装类命名为 Handler。既然作为消息队列的包装Handler肯定要持有MessageQueue，以便于向消息队列中插入消息。因此我们还需要对外提供插入消息的API，姑且将这个API命名为 sendMessage(Message msg),在这个方法内实现将 msg 插入到 MessageQueue 的逻辑。

三、消息机制框架实现亟需解决的问题

上一章中，我们只是简单搭建起了Android消息机制的框架，具体的逻辑实现却会面临着诸多的问题。例如消息机制重要的功能是线程切换，应该如何保证消息能被发送到特定的线程？系统无时无刻的都在产生 Message，如何解决 Message 频繁创建导致的内存抖动问题？像View的渲染、屏幕的刷新、Activity生命周期调用这些消息具有较高的优先级，需要优先执行应该如何处理？

1.如何保证消息能被发送到指定的线程？

消息机制一个很重要的需求，就是在子线程中获取到的数据需要发送给主线程进行页面UI的渲染。那应该如何保证消息能被发送到主线程呢？其实很简单，只需要在主线程去实例化”消费者” Looper 即可。前提条件是只需要让生产者Handler与消费者Looper持有同一个MessageQueue即可。这样不管Handler在哪个线程插入消息，最终都会由Looper在主线程中执行。

到此为止，我们面临的问题变成了如何保证Looper与Handler持有的是同一个MessageQueue？

细细分析一下我们上一章对Looper的设计似乎是有缺陷的，因为此时的Looper可以在主线程中实例化出多个对象，但是由于我们的需求是Handler必须与Looper持有同一个MessageQueue，显然主线程中能实例化多个Looper不符合这一需求。那如果我们硬性要求只能在主线程中实例化出一个Looper,并将这个Looper存储到主线程中，这样，在Handler初始化时就可以从主线程中取出唯一的Looper对象来共享Looper的MessageQueue了。这一方法确实能够实现Looper与Handler共享同一个MessageQueue。但是这个唯一的Looper应该怎么存储到主线程中，且不能被其他线程共享呢？

说到这里，java基础掌握好的同学应该已经想到了可以使用 ThreadLocal！ThreadLocal提供了线程级别的数据存储功能，即在A线程中使用 ThreadLocal 存储了一个数据，那么这个数据只对A线程可见，即只有在A线程中才能从取出，而其他线程则无法取到。关于 ThreadLocal 了解这么多就足够了，这里不再赘述ThreadLocal 的实现原理，如果你对ThreadLocal比较感兴趣可以参考我之前写的一篇文章《Java并发系列番外篇：ThreadLocal原理其实很简单》

2.如何设计消息与消息队列？

MessageQueue 中的消息绝大部分是 APP 在执行过程中由 Android 系统插入进去的，比如 Activity 启动时的生命周期时，以及事件派发过程程中都会向 MessageQueue 插入消息。这也就意味着会有大量的 Message 被实例化。如果每次实例化都通过new Message()来实现，那么势必会导致严重的内存抖动问题。除此之外，由于大量的Message消息，MessageQueue 应该如何设置容量也是需要考虑的问题。

（1）Message对象池

为了避免 Message 的频繁实例化，Android 系统中 Message 的实现使用了池化思想，也就是会有一个 Message 对象池，在 Message 用完之后会将其重置并将其放入到对象池中，等待下次复用。这样就避免了频繁实例化 Message 导致的内存抖动问题。这里直接贴一下简化后的源码实现。

可以看到 Message 实现对象池的两个核心方法就是 obtain()与 recycle()，obtain负责从对象池中取出Message，如果对象池没有空闲的 Message，则直接实例化 Message，而 recycle() 则是回收用完的Message，并将其插入到复用链表中。这里所谓的对象池其实就是一个空闲的 Message 链表。

（2）无界队列 MessageQueue

在上一小节中我们已经知道 Message 其实是一个拥有链表结构的类。因此MessageQueue中的容器其实并非像第二章第1小节中写的那样是一个LinkedList，而是一个Message链表。

另外，在第一章中”生产者-消费者“模型中举的例子 MessageQueue 的容量是特定的。而特定容量的容器必然需要考虑背压策略，就比如大家所熟知的 RxJava 由于内部使用的是有界队列，因此在生产者过于旺盛容量的情况下，容量不足的时候就会导致 MissingBackpressureException，而 Rxjava 也给出了多个背压策略，例如丢弃事件、扩容、或者直接抛出异常。

但是 Handler 中 MessageQueue 中的实现是一个无界队列，即插入消息永远不会被阻塞，所以也不用考虑背压策略的问题了。设计成无界队列的主要原因还是因为 message 都是重要的消息，不可能因为容量不足导致丢弃一些消息或者直接抛出异常，如果这样恐怕Android系统早就崩溃了。

3.消息的优先级

在 Android 系统中，View的绘制、事件分发以及Activity的生命周期都依赖于 Message 来执行。而这逻辑显然有更高的执行优先级，如果这些消息被前面的一些不重要的消息阻塞，必然会引起页面掉帧，或者阻塞Activity生命周期的执行，拖累Activity的启动速度，严重影响用户的使用体验。

如果让我们作为消息的设计者，一定会为这些系统级别的消息设置一个比较高的优先级，来保证这些消息优先执行。事实上，Android源码中的消息也确实存在优先级划分的。

就我的理解而言，消息一共被分为了四个优先级。优先级从高到低分别是异步消息、普通消息、IDleHandler 以及延迟消息。

可以看到在doTraversal方法中会将同步屏障移除消息掉，之后普通消息又会得到执行的机会。其实到这里也很容易理解为什么postSyncBarrier方法不允许开发者调用了，因为一旦开发者执行这个方法，且没有即使移除同步屏障就会导致普通消息再也没有被执行的可能。

（2）普通消息和延迟消息

关于普通消息和延迟消息其实没有什么可说的，普通消息的优先级比异步消息低毋庸置疑，而延迟消息由于在插入消息队列时会根据延迟时间确定插入到队列中的位置，即延迟越久的消息在队列中的位置越靠后。因此延迟消息的优先级是最低的。

消息机制的设计者设计 IdelHandler 的目的就是为了执行一些不那么紧急的任务，在异步消息和普通消息执行完后，处于空闲时间时才会开始执行 IdleHandler。

而 IdleHandler 在 Framework 的源码中也是被频繁用到的，比如典型的是 Activity 的 onDestroy 生命周期的回调就是通过向 MessageQueue 中添加 IdleHandler 来执行的，也就是说调用了Activity 的 finish 方法后并不会立即调用 onDestory 方法，而是要等到消息队列空闲时 onDestory 才会被执行。

那么此时就有个疑问，如果在Activity调用 finish 之后就不断的向 MessageQueue 中插入消息，会不会导致 Activity 的 onDestory 一直不会被调用呢？其实理论上是这样的，但是在系统的源码中做了一个兜底，即如果十秒后 Activity 依然没有被销毁则会主动调用 Activity 的 onDestory 来执行销毁逻辑。

关于 onDestory 部分的源码分析可以参考路遥的一篇文章《面试官：为什么 Activity.finish() 之后 10s 才 onDestroy ？》
